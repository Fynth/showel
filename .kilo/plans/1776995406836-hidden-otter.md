# Plan: Make Shovel Better Than DBeaver

## Executive Summary

Shovel already has a solid foundation: native Rust/Dioxus desktop app, 4 database backends, AI/ACP integration, editable result grids, and modular architecture. To surpass DBeaver, the focus should be on **closing feature gaps** where DBeaver is strong, while **doubling down on areas where Shovel already leads** (AI-native, modern UI, fast native performance).

---

## Phase 1: Critical Feature Gaps (Highest Impact)

These are the features DBeaver users expect immediately. Without them, migration from DBeaver is unlikely.

### 1.1 Export/Import Wizard UI
**Status:** `query-io` has the backend logic, but no UI exists.
- Add an "Export" button to the result table toolbar
- Create `ExportWizard` modal component with:
  - Format selection (CSV, JSON, XLSX)
  - Column picker (select/deselect columns)
  - Row scope (all, filtered, selected)
  - File path picker via `rfd`
  - Encoding options
  - Preview of first N rows
- Create `ImportWizard` modal component with:
  - File picker
  - Format auto-detection
  - Column mapping (source -> target table columns)
  - Preview + validation
  - Conflict resolution (skip, overwrite, error)
- Wire into workspace toolbar and explorer context menu
- **Files:** new `ui/src/screens/workspace/components/export_wizard.rs`, `import_wizard.rs`

### 1.2 DDL Generation
**Status:** Explorer has `describe_table` which fetches CREATE statements, but no "Generate DDL" action.
- Add "Generate DDL" to explorer context menu on tables, views, indexes
- Support generating: CREATE TABLE, CREATE INDEX, CREATE VIEW, INSERT statements (for data)
- Show DDL in a new read-only query tab with syntax highlighting
- Support full schema DDL export (all tables in a schema)
- **Files:** new `query-core/src/ddl_gen.rs`, explorer context menu updates

### 1.3 ER Diagram from Live Schema
**Status:** `er_diagram.rs` component exists (450 lines) but is not wired up.
- Wire the existing ER diagram component to auto-generate from explorer data
- Use foreign key metadata already available in `describe_table` to draw relationships
- Add "ER Diagram" button in workspace toolbar / explorer context menu
- Support per-schema and full-database diagram generation
- Interactive: click table -> navigate to structure, drag to rearrange
- Auto-layout using a force-directed or hierarchical algorithm
- **Files:** update `er_diagram.rs`, add auto-layout logic, wire into workspace

### 1.4 Stored Procedure / Function Browser
**Status:** Completely absent for all backends.
- Add `ExplorerNodeKind::Procedure`, `ExplorerNodeKind::Function` to explorer tree
- Backend implementations:
  - **PostgreSQL:** `pg_proc` + `information_schema.routines`
  - **MySQL:** `information_schema.routines`
  - **ClickHouse:** No native stored procs, skip
  - **SQLite:** No stored procs, skip
- Show procedure source code in a query tab
- Add "Execute Procedure" dialog with parameter input fields
- Add "Create Procedure" template in SQL editor
- **Files:** update `explorer/src/{postgres,mysql}.rs`, `models/src/explorer.rs`, explorer UI

### 1.5 Transaction Management
**Status:** All queries auto-commit. No BEGIN/COMMIT/ROLLBACK.
- Add transaction state to `ConnectionSession` model
- Add transaction toolbar buttons: BEGIN, COMMIT, ROLLBACK
- Visual indicator when a transaction is active (colored status bar)
- Auto-rollback on session disconnect with active transaction
- Support for SQLite, PostgreSQL, MySQL (ClickHouse has limited transaction support)
- **Files:** `models/src/app.rs`, `query-core/src/lib.rs`, workspace toolbar

### 1.6 Connection Test Button
**Status:** Connection test happens only on full connect, no standalone test.
- Add "Test Connection" button to connection forms
- Test without creating a session: connect, run `SELECT 1`, disconnect
- Show success/failure with latency and server version
- **Files:** `connection/src/lib.rs` (add `test_connection`), connect screen forms

## Phase 2: Competitive Features (Match DBeaver)

These bring Shovel to feature parity with DBeaver for most daily workflows.

### 2.1 Visual Query Builder
**Status:** Completely absent. DBeaver's most requested feature.
- Drag-and-drop table addition from explorer
- Visual table box with column checkboxes
- Join lines between tables (inferred from FK or user-drawn)
- WHERE clause builder with operators
- SQL preview panel showing generated query
- Bidirectional: edit SQL -> update visual, edit visual -> update SQL
- Start simple: single-table SELECT with columns, WHERE, ORDER BY, LIMIT
- Phase later: JOINs, GROUP BY, HAVING, subqueries
- **Files:** new `ui/src/screens/workspace/components/query_builder/` module

### 2.2 Database Object Management
**Status:** Only CREATE TABLE and DROP/TRUNCATE exist.
- **ALTER TABLE:** ADD COLUMN, DROP COLUMN, RENAME COLUMN, CHANGE TYPE
- **CREATE/DROP INDEX:** From explorer context menu
- **CREATE/DROP VIEW:** From explorer context menu or SQL editor
- **CREATE/DROP SCHEMA** (PostgreSQL, MySQL)
- **CREATE/DROP DATABASE** (PostgreSQL, MySQL, ClickHouse)
- **Rename table** (ALTER TABLE RENAME TO)
- **Files:** extend `query-core/src/ddl.rs`, explorer context menus

### 2.3 Schema Comparison / Migration Tool
**Status:** `data_diff.rs` component exists (403 lines) but only for row-level data.
- Add schema diff: compare table structures between two schemas or databases
- Show side-by-side DDL differences
- Generate migration SQL (ALTER TABLE statements to sync)
- Support comparing across different database types (limited)
- **Files:** new `query-core/src/schema_diff.rs`, reuse `data_diff.rs` UI patterns

### 2.4 Data Diff Integration
**Status:** Component exists but not wired up.
- Wire `data_diff.rs` into the UI with a "Compare Data" action
- Support: table vs table, query vs query, table vs query
- Column mapping interface
- Configurable comparison keys
- **Files:** wire existing component into workspace

### 2.5 Saved Queries with Folders/Tags
**Status:** Flat list only, no organization.
- Add folder support to `SavedQuery` model (optional `folder` field already exists)
- Folder tree in saved queries panel
- Drag-and-drop to organize
- Tag/filter system
- Search within saved queries
- **Files:** `models/src/saved_query.rs`, `storage/src/saved_queries.rs`, saved queries panel UI

### 2.6 Connection Groups
**Status:** Flat list of saved connections, max 10.
- Remove the 10-connection limit
- Add folder/group hierarchy to saved connections
- Drag-and-drop to organize
- Color-coded labels
- **Files:** `models/src/connection.rs`, `storage/src/history.rs`, connect screen UI

### 2.7 BLOB/CLOB Viewer Enhancement
**Status:** `blob_viewer.rs` exists (347 lines) with hex/text/image modes.
- Add download button to save BLOB to file
- Add upload button to insert file as BLOB
- Support large BLOBs with streaming/pagination
- Support audio/video preview for media types
- **Files:** extend `blob_viewer.rs`

### 2.8 Full-Text Search in Results
**Status:** Only column-rule filtering exists, no global search.
- Add a global search box above the result table
- Search across all visible columns simultaneously
- Highlight matching cells
- Support regex mode
- **Files:** `result_table.rs` or a new filter component

## Phase 3: Surpass DBeaver (Shovel's Unique Advantages)

These features leverage Shovel's unique strengths to go beyond what DBeaver offers.

### 3.1 AI-Powered Features (DBeaver has nothing comparable)
- **Natural Language to SQL:** Already partially working via ACP, but add a dedicated "Ask AI" button in the query toolbar
- **Query Explanation:** Select a query -> "Explain in Natural Language" -> AI describes what it does
- **Schema Suggestions:** AI detects missing indexes, normalization issues, and suggests improvements
- **Auto-Complete++:** Enhance CodeStral inline completion with schema-aware context (table names, column names, join suggestions)
- **Query Optimization:** "Optimize This Query" -> AI suggests rewrites with explanations
- **Data Anomalies:** "Find Anomalies" -> AI scans table data for outliers, NULL patterns, duplicates
- **Files:** extend ACP context, add AI action buttons throughout the UI

### 3.2 Smart Schema Explorer
- Inline row counts in the tree (lazy-loaded, cached)
- Quick stats on hover: row count, size, last modified
- Search/filter the tree with regex
- "Recent Tables" section showing most-accessed tables
- "Favorites" for pinning frequently used tables
- **Files:** explorer crate + UI updates

### 3.3 Advanced Execution Plan Visualization
**Status:** `execution_plan.rs` exists (255 lines) with tree/table views.
- Add visual flame-graph style rendering
- Color-code expensive nodes (red = slow, green = fast)
- Side-by-side comparison of two execution plans (before/after optimization)
- AI-powered plan explanation
- **Files:** extend `execution_plan.rs`

### 3.4 Database Health Dashboard
**Status:** `introspection.rs` has all the backend data (locks, active queries, slow queries, index stats).
- Create a "Dashboard" tab showing:
  - Active connections / queries with kill button
  - Lock graph / blocking chain
  - Index usage statistics (unused indexes, missing indexes)
  - Table size growth trends
  - Slow query log
- Real-time auto-refresh with configurable interval
- Alert thresholds (e.g., "queries running > 30s")
- **Files:** new `ui/src/screens/workspace/components/dashboard.rs`, wire introspection data

### 3.5 Cross-Database Queries / Data Transfer
- Copy table data between different database backends
- Schema migration wizard (PostgreSQL -> MySQL, etc.)
- Type mapping for cross-database transfers
- Progress tracking with row count
- **Files:** new `query-core/src/data_transfer.rs`

### 3.6 Keyboard-First Workflow
- Configurable keyboard shortcuts for all actions
- Command palette (Ctrl+Shift+P style) for quick action access
- Vim-mode option for SQL editor
- Quick navigation: Ctrl+T for table search, Ctrl+E for recent queries
- **Files:** new `ui/src/hooks/use_keyboard_shortcuts.rs`, command palette component

## Phase 4: Architecture & Quality Improvements

These don't add features but make the codebase sustainable for all the above work.

### 4.1 Decompose Monolithic Components
- Split `ui/src/screens/workspace/mod.rs` (917 lines) into focused submodules
- Split `ui/src/screens/workspace/components/result_table.rs` (1872 lines) into:
  - `result_grid.rs` (rendering)
  - `result_filters.rs` (filter UI + logic)
  - `result_editing.rs` (inline edit logic)
  - `result_row_details.rs` (row details panel)
- Split `query-core/src/lib.rs` into more focused modules
- **Priority:** Do this BEFORE adding new features to these files

### 4.2 Reduce Prop Drilling
- Move more workspace state into Dioxus contexts (already has 3, needs ~8-10)
- Create `WorkspaceLayoutContext`, `WorkspaceExplorerContext`, `WorkspaceActionsContext`
- This unblocks cleaner feature additions

### 4.3 Error Handling & Recovery
- Add retry mechanism for failed connections
- Detailed error panel with copy-to-clipboard and "Search Error" button
- Graceful degradation when partial operations fail
- Connection health monitoring with auto-reconnect

### 4.4 Testing Infrastructure
- Add integration tests for each driver
- Add snapshot tests for DDL generation
- Add UI component tests where possible
- Target: >70% coverage on `query-core`, `explorer`, `connection`

### 4.5 Accessibility
- ARIA attributes for tree views, modals, and interactive elements
- Keyboard navigation for explorer tree
- Focus management for modal open/close
- Screen reader announcements for toast notifications

## Sprint 0: i18n Infrastructure (MUST be first)

Everything else depends on this because every new UI string needs to be translatable from the start.

### 0.1 i18n Crate
- Create `i18n` workspace crate for localization infrastructure
- Use `fluent` (Mozilla's localization system, pure Rust) or `rust-i18n` crate
- Locale files in `i18n/locales/{en,ru}/` as TOML or Fluent files
- Fallback chain: requested locale -> English
- Detect system locale on startup, persist override in `AppUiSettings`

### 0.2 i18n Model & Storage
- Add `language: String` field to `AppUiSettings` (default: `"en"`)
- Add language selector to settings modal
- Load locale files at startup, provide via Dioxus context or global signal

### 0.3 i18n Macro / Hook
- Create `t!("key")` macro or `use_i18n()` hook for Dioxus components
- Support interpolation: `t!("greeting", name = user.name)`
- Support plural forms
- Wrap all existing hardcoded strings in the UI with `t!()` calls
- **Files:** new `i18n/` crate, update `models/src/settings.rs`, `ui/src/app.rs`, all UI components

### 0.4 Existing String Extraction
- Extract all hardcoded strings from UI components into locale files
- Start with English (`en`) and Russian (`ru`) locales
- ~300-500 strings estimated across the current UI
- **This is the most labor-intensive part** but only needs to be done once
